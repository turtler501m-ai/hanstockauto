# AI 자동매매 시스템 구현 리스크 및 한계 분석

본 문서는 `ai-trading-system-plan.md` 기획 및 신규 AI 대시보드 구조를 실제 코드로 구현하고 운영할 때 예상되는 **기술적 한계(어려움)**와 **구현 계획상의 문제점**, 그리고 이를 극복하기 위한 **해결 방안**을 분석합니다.

---

## 1. 인프라 및 아키텍처 문제 (GitHub Actions 한계)

### 🔴 문제점
현재 한스톡 시스템은 **GitHub Actions의 Cron 스케줄링**을 메인 실행 엔진으로 사용하고 있습니다. 하지만 진정한 AI 시스템(모델 추론, 워크포워드 검증, 실시간 리스크 검사)을 돌리기에는 다음과 같은 치명적 단점이 있습니다.
1. **상태 유지 불가(Stateless)**: GitHub Actions는 매번 가상 머신이 초기화됩니다. 의사결정 로그(Decision Log)나 SQLite DB, 모델 파일(.pkl)을 매번 업로드/다운로드해야 하므로 속도 저하와 데이터 유실 위험이 매우 큽니다.
2. **실시간 대응 불가**: 지정된 스케줄(예: 2시간 간격)에만 깨어나므로, 장중 급락 사태 발생 시 대시보드의 `Kill Switch`를 눌러도 다음 스케줄 전까지는 즉각적인 대응(전량 매도 등)이 어렵습니다.
3. **무거운 AI 모델 추론**: LightGBM, PPO 등의 모델을 매번 설치하고 메모리에 올리는 데(Cold Start) 긴 시간이 소요됩니다.

### 🟢 해결 방안
* **단기**: 현재의 GitHub Actions 구조를 유지하되, DB와 모델 가중치 파일은 외부 클라우드 스토리지(AWS S3, Google Cloud Storage)나 별도의 원격 DB(Supabase, PlanetScale 등)에 연동하여 상태를 안전하게 영속화해야 합니다.
* **장기**: AI 추론과 리스크 엔진을 전담하는 **상시 기동 서버(Always-on VPS, AWS EC2 등)** 위에서 스케줄러(APScheduler, Celery 등)로 구동하도록 아키텍처를 마이그레이션해야 진정한 관제센터와 실시간 Kill Switch 연동이 가능합니다.

## 2. 데이터 정합성 (Online-Offline 불일치)

### 🔴 문제점
AI 매매에서 가장 흔하게 겪는 실패 원인은 "백테스트에서는 수익이 났는데 실전에선 박살 나는 현상(Lookahead Bias 및 Data Leakage)"입니다.
1. **과거 데이터(Offline)와 실시간 데이터(Online)의 차이**: 모델을 학습시킬 때 사용한 증권사 덤프 데이터와, 실시간 KIS API로 받아오는 데이터(분봉, 호가 등)의 지표값(결측치, 수정주가 반영 여부 등)이 미세하게 다를 수 있습니다.
2. **Feature Generation의 파편화**: 과거 백테스트 시 피처(RSI, MACD 등)를 뽑는 코드와 현재 `trader.py`에서 피처를 뽑는 코드가 분리되어 있으면, 모델은 자신이 학습하지 않은 엉뚱한 값을 실전에서 보고 판단을 내리게 됩니다.

### 🟢 해결 방안
* **Feature Store 도입**: 과거 학습용 피처 생성 함수와 실전 추론용 피처 생성 함수를 무조건 **단일 모듈(`feature_pipeline.py`)**로 통일해야 합니다.
* 모의 투자(Demo/Paper) 환경에서 일정 기간 모델의 **예측값(Score)과 실제 결과를 로깅**하여, 백테스트 당시의 예측값 분포와 실전 예측값 분포가 일치하는지 먼저 검증해야 합니다.

## 3. 의사결정 로그(Decision Log) 비대화 문제

### 🔴 문제점
기획서에 명시된 "모든 주문 판단이 재현 가능해야 한다"는 목표를 달성하려면, 매 스캔 시점마다 모든 후보 종목의 피처 10~20개, 모델 예측 점수, 리스크 검사 결과를 모두 저장해야 합니다.
1. **DB 용량 폭발**: 종목 100개를 하루 10번 스캔하면 매일 1,000개의 Decision Log가 발생하며, 한 달이면 수만 건이 쌓여 로컬 SQLite의 성능을 크게 저하시킵니다.

### 🟢 해결 방안
* **계층형 로깅 전략**: 
  * 주문이 실제로 발생했거나, 승인 큐에 올라간 '유의미한' 후보들에 대해서만 Full Feature를 DB에 저장합니다.
  * 단순히 스캔 단계에서 탈락한 종목들은 경량화된 로그(.csv 또는 JSON Lines) 형태로 파일로 남겨 외부에 백업합니다.

## 4. 리스크 엔진과 상태 관리 (Paper vs Live)

### 🔴 문제점
대시보드 기획에는 `Paper 성과`와 `총 평가자산(Live)`을 동시에 비교하는 기능이 있습니다.
1. **가상 원장(Ledger) 동기화의 복잡성**: KIS 증권사의 실계좌 잔고는 1개뿐입니다. 프로그램 내부에서 "만약 A모델을 썼더라면(Paper)"의 장부와 "실제 내 계좌(Live)" 장부를 동시에 추적하려면 거래 기록 체계가 완전히 분리되고 복잡해집니다.
2. 이전 문제처럼 동기화(`sync_trades`) 과정에서 가상 장부와 실제 장부가 꼬이게 되면 수익률 통계가 완전히 무너질 수 있습니다.

### 🟢 해결 방안
* **DB 스키마 분리**: `trades` 테이블 하나로 관리하지 말고, `live_trades`, `paper_trades` 테이블을 물리적으로 분리하거나 `portfolio_type='LIVE' | 'PAPER'` 컬럼을 추가해 철저히 격리해야 합니다.
* 실전(Live) 계좌 동기화는 **'절대적 진실(Source of Truth)'**로 KIS 잔고를 믿되, Paper 계좌는 독립된 시뮬레이션 환경으로 동기화 로직에서 배제해야 합니다.

## 5. 강화학습(RL) 및 PPO 모델 적용의 함정

### 🔴 문제점
FinRL 등 강화학습을 적용하는 것은 이론상 훌륭하나, 주식 시장의 체제 변화(Regime Shift - 횡보장 vs 폭락장)에 매우 취약합니다.
1. 어제까지 잘 작동하던 PPO 모델이 오늘 폭락장에서는 "계속 홀딩"이라는 엉뚱한 판단을 내릴 수 있습니다. 모델의 결과가 왜 그렇게 나왔는지 설명 불가능(Black box)하기 때문에 수동 통제가 어렵습니다.

### 🟢 해결 방안
* 기획서에도 나와 있듯 **Rule-based 리스크 엔진을 모델보다 우위에 두어야 합니다.**
* AI 모델(PPO, LGBM)이 "매수 비중 50%"를 제안하더라도, 리스크 엔진이 "현재 변동성 초과"라고 판단하면 강제로 비중을 0%로 깎아버리는(Override) 구조가 시스템적으로 하드코딩 되어 있어야 합니다. 대시보드의 '주문 라우터' 파이프라인에서 AI Score 통과 후 Risk 단계에서 차단되는 흐름이 이를 해결합니다.

## 종합 결론 및 우선순위 조정

기획안은 매우 이상적이고 완성도가 높지만, 이를 현재의 단일 `trader.py` + GitHub Actions 구조 위에 무작정 얹으면 시스템이 붕괴될 위험이 큽니다.

따라서 다음과 같은 순서로 기반 공사를 먼저 진행할 것을 권장합니다.
1. **아키텍처 이전 준비**: GitHub Actions에 의존하는 상태 저장(DB, Model)을 외부 클라우드나 원격 DB로 이전.
2. **분리된 피처 파이프라인 구축**: 모델 추론 코드와 별도로, 과거/현재 데이터를 동일하게 가공하는 모듈(`features.py`) 분리.
3. **리스크 엔진 독립화**: `trader.py` 안에 섞여 있던 방어 로직들을 독립된 `RiskEngine` 클래스로 뽑아내어, 모든 매수/매도 신호가 무조건 이 클래스를 거치게 리팩토링.
4. **AI 모델 연동 및 대시보드 API 확장**: 그 이후에 신규 대시보드 화면에 맞춘 FastAPI 엔드포인트를 구현.
