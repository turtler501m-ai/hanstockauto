# Seven Split Auto Trading 프로젝트 개선 제안서

현재 코드(`src/trader.py` 및 전체 아키텍처)를 분석한 결과, 시스템의 안정성, 유지보수성, 확장성을 높이기 위한 몇 가지 개선안을 제안합니다.

## 1. 아키텍처 및 모듈화 (Refactoring)
현재 `trader.py` (약 1,300줄 이상) 파일 하나에 API 통신, 트레이딩 전략, 데이터베이스 처리, Slack 알림 등 너무 많은 로직이 혼재되어 있습니다. 이를 기능별로 분리하여 **단일 책임 원칙(SRP)**을 준수해야 합니다.
* **제안하는 구조:**
  * `src/api/kis_api.py`: 한국투자증권 API 통신 및 인증, 서킷브레이커 로직.
  * `src/strategy/seven_split.py`: RSI, MACD, Bollinger Bands 등 지표 계산 및 세븐 스플릿 매매 로직.
  * `src/db/repository.py`: SQLite 데이터베이스 연결 및 CRUD.
  * `src/notifier/slack.py`: Slack 메시지 전송 및 알림 포맷팅.
  * `src/main.py`: 설정 로드 및 전체 워크플로우 오케스트레이션.

## 2. 설정 관리 (Configuration Management)
환경변수(`os.environ`)를 코드 여러 곳에서 직접 호출하여 사용 중이며, 타입 캐스팅(`int()`, `float()`)이 파편화되어 있습니다.
* **해결 방안:** `pydantic-settings` 라이브러리를 도입하여 애플리케이션 시작 시 모든 환경 변수를 검증(Validation)하고 힌팅(Type Hinting)을 제공하는 싱글톤 설정 객체(`config.py`)를 운영합니다.

## 3. 로깅 시스템 고도화 (Logging System)
현재의 로깅 방식은 `print`와 `open().write()`를 통한 수동 로깅입니다. 이는 멀티스레드 환경이나 로그 파일의 크기가 커질 때 병목이나 파일 손상을 일으킬 수 있습니다.
* **해결 방안:** Python 표준 `logging` 모듈 또는 `loguru` 라이브러리를 도입하여 **비동기 로깅, 로그 포맷팅, 로그 파일 로테이션(크기/날짜 단위)** 기능을 추가해야 합니다.

## 4. 백테스팅(Backtesting) 환경 구축
실전 투자를 위한 3단계 방어 로직(Dry Run 등)은 훌륭하지만, 과거 데이터를 기반으로 봇의 수익률을 검증할 독립적인 백테스팅 프레임워크가 부족합니다.
* **해결 방안:** `Backtrader`나 `zipline`과 같은 파이썬 백테스트 라이브러리를 연동하거나, 기존에 구축된 `rl_env.py`(강화학습 환경)를 확장하여 과거 데이터 주입 시뮬레이터 클래스를 작성하는 것을 권장합니다.

## 5. 의존성 및 패키지 관리
`requirements.txt`로 관리되고 있으나, 의존성 간의 버전 충돌을 막기 위해 현대적인 패키지 매니저 도입을 고려할 수 있습니다.
* **해결 방안:** `Poetry` 또는 `uv` 등의 툴을 사용하여 `pyproject.toml` 기반으로 의존성을 관리하고 Lock 파일을 생성하여 배포 환경의 일관성을 높입니다.

## 6. 예외 처리 및 서킷 브레이커 개선
현재 클래스 변수로 상태를 관리하는 `_check_circuit()` 방식은 앱 재시작 시 상태가 초기화되는 한계가 있습니다.
* **해결 방안:** `tenacity` 패키지를 활용한 데코레이터 패턴 도입 혹은, 서킷 브레이커 상태를 Redis나 SQLite 등 외부 스토리지에 기록하여 프로세스 재시작에도 내결함성을 유지하도록 합니다.
