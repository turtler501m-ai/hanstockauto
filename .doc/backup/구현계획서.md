# Seven Split Auto Trading 프로젝트 구체적 구현 계획서

제안된 개선안을 바탕으로 실제 코드에 적용하기 위한 단계별(Phase) 구현 계획입니다. 리스크를 최소화하기 위해 점진적 리팩토링 방식을 취합니다.

---

## 🚀 Phase 1: 기반 인프라 재구축 (설정 및 로깅)
가장 먼저 시스템의 전역 상태(환경 변수, 로깅)를 체계화합니다. 이 작업은 다른 모듈 분리 작업의 기초가 됩니다.

**1. 설정 관리 (Configuration) 도입**
*   **작업 내용:** `pydantic-settings` 라이브러리를 사용하여 `src/config.py` 생성.
*   **세부 구현:**
    *   기존 `os.environ.get()` 형태를 모두 `pydantic.BaseSettings`를 상속받은 `Settings` 클래스로 마이그레이션.
    *   타입 힌팅(`int`, `float`, `bool`) 적용 및 필수 값 누락 시 앱 시작 단계에서 Fail-fast 발생.
*   **패키지 추가:** `pip install pydantic pydantic-settings`

**2. 로깅 고도화 (Logging)**
*   **작업 내용:** `loguru` 라이브러리를 도입하여 로깅 모듈화.
*   **세부 구현:**
    *   `src/utils/logger.py` 파일 생성 후 전역 로거 설정.
    *   로그 로테이션(예: 매일 자정 혹은 10MB 초과 시 분리) 및 보존 기간(예: 30일) 설정.
    *   기존 코드의 `log()` 함수와 `print()` 문을 `logger.info()`, `logger.error()` 등으로 전면 교체.
*   **패키지 추가:** `pip install loguru`

## 🛠 Phase 2: 아키텍처 모듈화 (Refactoring)
비대해진 `src/trader.py`를 단일 책임 원칙(SRP)에 따라 도메인별 디렉토리로 분리합니다.

**1. 디렉토리 구조 재편**
```text
src/
├── api/
│   ├── __init__.py
│   └── kis_api.py       # KIStockAPI 클래스 이동
├── db/
│   ├── __init__.py
│   └── repository.py    # sqlite3 연결 및 save_trade() 로직
├── notifier/
│   ├── __init__.py
│   └── slack.py         # send_slack(), slack_order() 등 알림 로직
├── strategy/
│   ├── __init__.py
│   ├── indicators.py    # calc_rsi, calc_macd 등 순수 수학 함수
│   └── seven_split.py   # 실제 매매 결정 및 스코어링 로직
├── utils/
│   └── logger.py
├── config.py            # Phase 1에서 만든 설정 파일
├── trader.py            # 메인 실행 스크립트 (오케스트레이션 역할만 수행)
└── dashboard.py         # 기존과 동일하게 유지 (import 경로만 수정)
```

**2. 순차적 분리 진행**
*   **1단계 (독립 모듈 분리):** 가장 의존성이 적은 `db/repository.py`와 `notifier/slack.py`, `strategy/indicators.py`부터 분리.
*   **2단계 (API 모듈화):** `api/kis_api.py`를 분리하고 `config.py`와 `logger.py`를 주입받아 사용하도록 수정.
*   **3단계 (전략 분리):** 매매 로직을 `strategy/seven_split.py`로 묶어 API 인스턴스와 DB 인스턴스를 주입받아 동작하게 함.
*   **4단계 (통합):** `src/trader.py`를 메인 진입점(Entrypoint)으로 수정. 각 모듈을 인스턴스화하여 스케줄에 맞게 호출하는 역할만 하도록 축소.

## 🛡 Phase 3: 내결함성(Resiliency) 강화
API 호출 실패나 네트워크 불안정성에 대비하여 서킷 브레이커와 재시도 로직을 강화합니다.

**1. 재시도(Retry) 로직 도입**
*   **작업 내용:** `tenacity` 라이브러리를 사용해 API 호출 실패 시 자동 재시도 로직 구성.
*   **세부 구현:**
    *   `kis_api.py` 내의 HTTP 요청(ex. `get_balance`, `place_order`) 메소드에 `@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))` 데코레이터 적용.
    *   이를 통해 기존 수동으로 작성된 재시도 루프(for attempt in range(1, 3))를 제거하고 코드 가독성 향상.
*   **패키지 추가:** `pip install tenacity`

**2. 서킷 브레이커 영속성 확보**
*   기존의 메모리 기반 변수(`cls._err_count`) 대신 스크립트가 크론(Cron)으로 재시작되어도 상태를 유지할 수 있도록 SQLite(`db/repository.py`)나 별도의 캐시 파일(json)에 서킷 브레이커 상태를 저장.

## 📈 Phase 4: 백테스팅(Backtesting) 및 의존성 관리 개선
코드가 모듈화된 후 추가적인 검증 환경을 구축합니다.

**1. 패키지 매니저 마이그레이션**
*   `requirements.txt` 대신 `uv` 또는 `Poetry`를 도입해 `pyproject.toml`로 의존성을 관리.

**2. 백테스팅 프레임워크(선택 사항)**
*   `src/strategy/seven_split.py`로 분리된 로직은 의존성이 분리되었으므로, KIS API 대신 모의 `BacktestAPI` 클래스를 주입하여 `yfinance`에서 과거 데이터를 받아와 가상 매매를 수행하는 `src/backtest.py` 구현.
